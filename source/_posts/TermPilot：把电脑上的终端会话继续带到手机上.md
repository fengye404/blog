# TermPilot：把电脑上的终端会话继续带到手机上

> 本文由 AI 参与创作

## 前言

最近我在做 `TermPilot`。它要处理的是一个很具体的问题：电脑上已经跑起来的一条终端会话，人离开工位之后，手机还能不能继续接上。

这篇先把项目本身讲清楚。后面那篇再讲仓库是怎么围绕 agent 做 harness engineering 的。因为如果项目到底在解决什么问题、边界收在哪里都没先说明白，后面很多关于 `AGENTS.md`、运行时边界、验证和 CI 的讨论其实都悬着。

## 1. 我到底想解决什么问题

我遇到的场景很稳定，基本都长这样：

- 电脑上跑着 `Claude Code`
- 或者跑部署脚本、数据库 migration、长时间批处理
- 人离开工位了，但任务还没结束
- 手机上主要想继续看输出，必要时补一条命令，或者直接中断

真正麻烦的地方是，上下文已经在原来的终端里了：

- 当前目录已经切好了
- 前面的输出已经滚出来了
- 命令已经执行到一半了
- 有些确认、交互和中断语义也都挂在这条会话上

所以 `TermPilot` 的重点从一开始就是把原来那条会话保下来。很多现成方案能解决“重新进入机器”，但默认路径还是新建连接、新建上下文。对我来说，这一步差得其实很大。

如果把它压成一句话，这个项目做的就是“会话连续性”。

## 2. 当前实现怎么收

当前实现分成三个 runtime：

- `relay`：部署在一台手机能访问到的机器上，负责 HTTP 入口、配对、授权、审计元数据和 WebSocket 中转
- `agent`：运行在你自己的电脑上，负责本地会话、`tmux` 生命周期、输出回放和状态同步
- `app`：移动端 Web 页面，由 `relay` 直接托管

会话后端现在用的是 `tmux`。原因也很直接：我要的是一条可以附着、分离、继续查看的长期会话，而不是一次性的命令执行器。

这里有一条我刻意收得比较硬的边界：session 的 source of truth 在 `agent`，不在 `relay`。当前仓库里，`relay` 默认只持久化 pairing、grant、audit 这类最小元数据，长期运行默认也是 SQLite；会话标题、cwd、状态细节和终端输出都留在电脑侧。

服务器端先启动 `relay`：

![relay 启动截图](./relay-start.png)

电脑端启动 `agent` 后，会打印设备 ID、设备指纹和一次性配对码，手机浏览器用这组信息完成绑定：

![agent 启动与配对码截图](./agent-pairing.png)

这组拆法背后的工程判断也比较明确：

- `relay` 只做公网入口、配对和转发，不承载终端主数据
- `agent` 才是本地会话的真实持有者
- `app` 负责查看、轻输入和轻控制，不吸收后端语义

## 3. 主路径怎么跑通

如果只看主路径，基本就是下面几步：

```bash
npm install -g @fengye404/termpilot

# 服务器
termpilot relay

# 电脑
termpilot agent --relay wss://your-domain.com/ws
termpilot claude code
```

如果不是 `Claude Code`，也可以直接跑别的托管命令：

```bash
termpilot run -- <command>
```

关键点在最后这一步。无论是 `termpilot claude code`，还是 `termpilot run -- <command>`，本质上都是在电脑上创建一条受管理的本地 `tmux` 会话，然后让桌面和手机都接到同一条上下文上。

比如桌面端进入 `Claude Code` 后，安全确认仍然发生在原来的终端上下文里：

![Claude Code 安全确认截图](./claude-trust-folder.png)

手机端看到的就是这条会话本身的输出和状态：

![手机端会话截图](./mobile-session.png)

我比较在意的是这条路径的一致性：

- 输出来自同一条本地会话
- 控制动作写回同一条本地会话
- 会话退出也跟着这条受管理会话自己的语义走

这也是为什么我没有去做“手机重新开一个壳，再想办法补上下文”那条路。那样能连上机器，但连上的不一定还是同一条任务。

## 4. 手机端为什么只做轻控制

手机端当前做的事情比较克制，主要有四类：

- 看会话列表和当前状态
- 查看最新终端输出
- 补一条短输入或者发快捷键
- 在需要的时候切到专注模式

这个取舍是有意的。`TermPilot` 里的手机端更像一个任务面板，当前也不承担长期重度终端编辑。它要解决的就是“离开桌面之后怎么继续跟住这条任务”。

所以当前页面会有：

- 短输入
- 终端键盘
- 常用快捷键
- 专注模式

这些能力已经足够支撑查看、补命令和轻控制。另外仓库里也补了后台提醒相关能力，比如设备离线、会话退出和疑似残留会话的提醒。它们都是围绕“跟住这条任务”服务的，没有继续往手机重度编辑那条方向扩。

## 5. 这版边界收在哪里

当前这版 `TermPilot` 没有打算覆盖下面这些方向：

- 远程桌面
- 图形界面控制
- 手机上长时间重度终端编辑
- 自动接管任意已有 Terminal / iTerm 标签页
- 通用多租户运维平台

尤其是“自动接管任意已有 Terminal / iTerm 标签页”这一点，仓库首页其实也写得很明确：只有由 `TermPilot` 创建或管理的会话，手机端才能继续访问。

这些不是功能还没补完，而是产品边界本来就收在这里。边界一旦放大，下面这些设计都得一起重做：

- `relay` 要不要继续只存最小元数据
- `app` 要不要承担更重的交互和编辑负担
- 会话语义是不是还围绕同一条本地终端
- 设备授权、同步和残留治理是不是还成立

对现在这个项目来说，我更想先把“会话连续性”这条主路径做顺。边界先收住，很多实现判断才能跟着收住。

## 6. 为什么后面还要写 harness engineering

把项目本身讲清楚之后，后面那篇 harness engineering 才有落点。因为仓库里后来补的那些工程结构，本质上都在服务前面这条产品路径：

- 为什么仓库入口要固定成 `AGENTS.md`、`ARCHITECTURE.md`、`PLANS.md`
- 为什么 runtime boundary 要落成明确约束
- 为什么验证要整理成仓库内可直接复跑的入口
- 为什么文档漂移也要进入检查

如果前面不先把 `TermPilot` 讲清楚，后面这些内容就很容易写成脱离项目的 agent 工程空话。

## 7. 结语

如果你平时确实会在电脑上挂着一条长期任务，比如 `Claude Code`、部署脚本、迁移或者批处理，然后离开桌面之后还想继续看它、补一点轻控制，那 `TermPilot` 这条路径会比较合适。

它的核心就是把一条已经在跑的终端会话继续保留下来。

这一篇先把项目本身讲清楚。后面那篇再讲它是怎么从“先把功能做出来”，逐步整理到更适合 agent 持续接手的状态。

# TermPilot：把电脑上的终端会话继续带到手机上

> 本文由 AI 参与创作

## 前言

最近我在做 `TermPilot`。它是一个终端会话接力工具，目标很收敛：电脑上已经跑起来的一条终端会话，在人离开工位之后，手机还能继续接上。

这篇先把项目本身讲清楚。后面那篇再讲仓库怎么围绕 agent 做 harness engineering。如果项目形态和边界没有先说清楚，后面很多关于 `AGENTS.md`、验证、CI 和 runtime boundary 的讨论都很容易失去落点。

## 1. 先把问题说清楚

我遇到的场景很稳定，基本都长这样：

- 电脑上跑着 `Claude Code`
- 或者跑部署脚本、数据库 migration、长时间批处理
- 人离开工位了，但任务还没结束
- 手机上主要想继续看输出，必要时补一条命令，或者直接中断

这里真正麻烦的地方在于，上下文已经在原来的终端里了：

- 当前目录已经切好了
- 前面的输出已经滚出来了
- 命令已经执行到一半了
- 这条会话当前是什么状态，桌面端最清楚

这也是我为什么没有把它往“再做一个远程登录工具”那个方向去做。很多现成方案能解决“重新进入机器”，但很难把“原来那条会话”作为默认路径保留下来。`TermPilot` 的重点就是把这条路径单独做出来。

## 2. 当前实现长什么样

当前实现分成三个 runtime：

- `relay`：部署在一台手机能访问到的机器上，负责 HTTP 入口、配对、授权和 WebSocket 中转
- `agent`：运行在你自己的电脑上，负责本地会话、输出抓取和状态同步
- `app`：移动端 Web 页面，由 `relay` 直接托管

会话后端目前用的是 `tmux`。这个选择很直接，因为我需要的是一条可以附着、分离、继续查看的长期会话，而不是一次性的命令执行器。

这里有一个运行时边界我刻意收得比较硬：session 的 source of truth 在 `agent`，不在 `relay`。`relay` 当前只保存 pairing、grant、audit 这类最小元数据；会话标题、终端输出、cwd、状态详情仍然保留在电脑侧。

服务器端先启动 `relay`：

![relay 启动截图](./relay-start.png)

电脑端启动 `agent` 后，会打印设备 ID、设备指纹和一次性配对码，手机浏览器用这组信息完成绑定：

![agent 启动与配对码截图](./agent-pairing.png)

从当前实现来看，这条路径的工程判断很明确：

- `relay` 负责入口和路由，不承载终端主数据
- `agent` 负责本地会话和输出同步
- `app` 负责查看、轻输入和轻控制

## 3. 一条会话怎么跑通

如果只看主路径，基本就是下面几步：

```bash
npm install -g @fengye404/termpilot

# 服务器
termpilot relay

# 电脑
termpilot agent --relay wss://your-domain.com/ws
termpilot claude code
```

最后这一步比较关键。`termpilot claude code` 会在电脑上创建一条受管理的本地 `tmux` 会话，然后把桌面和手机都接到这一条上下文上。

比如桌面端进入 `Claude Code` 后，安全确认仍然发生在原来的终端上下文里：

![Claude Code 安全确认截图](./claude-trust-folder.png)

手机端看到的就是这条会话本身的输出和状态：

![手机端会话截图](./mobile-session.png)

这条路径里我最在意的是一致性：

- 输出来自同一条本地会话
- 控制动作写回同一条本地会话
- 会话退出语义也跟着这条会话走

## 4. 手机端到底做什么

手机端当前做的事情比较克制，主要有四类：

- 看会话列表和当前状态
- 查看最新终端输出
- 补一条短输入或者发快捷键
- 在需要的时候切到专注模式

这个取舍是有意的。因为我想解决的是“离开桌面之后怎么继续跟住这条任务”。手机端当前不承担长期重度终端编辑。

所以当前页面会有：

- 短输入
- 终端键盘
- 常用快捷键
- 专注模式

这些能力足够支撑查看、补命令和轻控制，但不会假装自己已经变成完整桌面终端。

## 5. 当前边界

当前这版 `TermPilot` 没有打算覆盖下面这些方向：

- 远程桌面
- 图形界面控制
- 手机上长时间重度终端编辑
- 自动接管任意已有 Terminal / iTerm 标签页
- 通用多租户运维平台

这不是功能没来得及做，而是产品边界本来就收在这里。边界一旦放大，很多设计都会跟着变：

- `relay` 要不要继续只存最小元数据
- `app` 要不要承担更重的交互
- 会话语义是不是还围绕同一条本地终端
- 当前的同步方式和治理方式还够不够

对现在这个项目来说，我更希望先把“会话连续性”这条主路径做顺，再去讨论别的扩展方向。

## 6. 为什么后面还要写 harness engineering

把项目本身讲清楚之后，后面那篇 harness engineering 才有落点。因为像下面这些工程判断，都不是孤立存在的：

- 为什么仓库入口要固定成 `AGENTS.md`、`ARCHITECTURE.md`、`PLANS.md`
- 为什么 runtime boundary 要落成明确约束
- 为什么验证要整理成仓库内可直接复跑的入口
- 为什么文档漂移也要进入检查

这些东西最终都服务于前面这条产品路径，而不是脱离项目去空谈 agent 工程。

## 7. 结语

如果你平时确实会在电脑上挂着一条长期任务，然后离开桌面之后还想继续看它、补一点轻控制，`TermPilot` 这条路径会比较合适。

这一篇先把项目本身讲清楚。后面那篇再讲它是怎么从“先把功能做出来”，逐步整理到更适合 agent 持续接手的状态。

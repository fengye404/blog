# TermPilot：把电脑上的终端会话继续带到手机上

> 本文由 AI 参与创作

## 前言

前两天我写了一篇 [《从 Prompt Engineering 到 Harness Engineering：TermPilot 的改造实践》](https://fengye404.top/2026/03/22/%E4%BB%8EPrompt-Engineering%E5%88%B0Harness-Engineering%EF%BC%9ATermPilot%E7%9A%84%E6%94%B9%E9%80%A0%E5%AE%9E%E8%B7%B5/)。那篇文章的重点是工程改造，讲的是当 agent 持续参与开发之后，仓库入口、边界约束、验证和 CI 应该怎么整理。

但写完之后我发现，一个前提还没有单独讲清楚：`TermPilot` 到底是什么。

如果项目本身没有先说明白，后面很多关于 harness engineering 的讨论其实是悬空的。因为文档体系、检查器和验证脚本这些东西，最终都不是为了“整理而整理”，而是为了服务一条具体的产品路径。

所以这篇先单独把 `TermPilot` 讲清楚。

`TermPilot` 是我业余时间写的一个项目，解决的是一个很具体的问题：电脑上已经有一条终端会话在跑了，人离开工位之后，怎么在手机上继续接上这条原会话，而不是重新开一个新的 shell。

## 1. 它解决的不是“远程进机器”，而是“继续接上原会话”

很多现成方案都能让你远程进一台机器，但默认路径通常是：

1. 重新登录
2. 重新开一个 shell
3. 重新补上下文

这条路径不是不能用，但它并不贴近很多真实工作场景。因为很多时候，你真正关心的并不是“能不能再进一次机器”，而是：

- 原来那条任务现在还在不在跑
- 最新输出是什么
- 我能不能补一条命令
- 我能不能发一个中断或者直接停掉它

换句话说，问题的重点不是“再建立一个远程入口”，而是“怎么把原来那条上下文继续接上”。

`TermPilot` 想做的就是这件事。

这个场景我自己碰到很多次。比如电脑上跑着 `Claude Code`、部署脚本、数据库迁移，或者一条耗时比较长的批处理。任务还没结束，但人已经不在桌前。这个时候我通常不想在手机上重新开一条新终端，也不想重新 SSH、重新找目录、重新找前面的输出。我只想继续看原来那条会话现在跑到哪了。

## 2. 它当前适合什么场景

如果把当前版本的使用场景压缩一下，我觉得主要就是下面这类：

- 电脑上已经有一条长期任务在跑
- 人不一定一直坐在电脑前
- 手机端主要负责查看和轻控制

这里的“轻控制”包括：

- 补一条短命令
- 发快捷键
- 中断或关闭当前会话

它不追求把完整桌面终端体验硬塞到手机里，而是更像一个面向长期任务的移动观察面板。

这也是为什么我现在比较强调“会话连续性”这个词。`TermPilot` 的价值不在于把手机变成一台完整开发机，而在于让一条已经在电脑上运行的上下文，离开桌面之后仍然持续可达。

## 3. 当前运行形态

`TermPilot` 现在的运行形态比较简单，主要分成三块：

- `relay`：部署在一台手机能访问到的机器上，负责 HTTP 入口、配对、授权和 WebSocket 中转
- `agent`：运行在你自己的电脑上，负责本地 `tmux` 会话管理、输出抓取和状态同步
- `app`：移动端 Web 页面，由 `relay` 直接托管

这里有一个边界是我比较在意的：会话标题、状态详情和终端输出保留在 `agent` 所在电脑，`relay` 只保存配对、grant、audit 这类最小元数据。

这样做的好处很直接：

1. 公网入口可以收敛到一个 `relay`
2. 电脑自己不用额外暴露入站端口
3. 会话主数据不需要长期放到服务端

当前后端会话容器用的是 `tmux`。原因也很简单：这类长期任务天然适合放在 `tmux` 里，它的附着、分离、恢复语义也比较清楚。

## 4. 实际怎么用

如果只看主路径，整个流程就是下面几步。

先在一台手机能访问到的机器上启动 `relay`：

```bash
npm install -g @fengye404/termpilot
termpilot relay
```

然后在自己的电脑上启动 `agent`：

```bash
termpilot agent --relay wss://your-domain.com/ws
```

启动成功后，终端会打印一次性配对码。手机浏览器打开 relay 页面，输入这个配对码，就能完成设备绑定。

接着，在电脑上启动一条受管理会话：

```bash
termpilot claude code
```

或者：

```bash
termpilot run -- python -m http.server
```

这时候桌面和手机接入的就是同一条会话。电脑上继续跑原程序，手机上继续看输出；如果需要，也可以在手机上补一条短命令、发一个快捷键，或者切到专注模式看最新输出。

![TermPilot 移动端页面截图](./termpilot-mobile-demo.png)

这个路径里我比较看重的一点是：手机端并不是重新开了一个“类似的环境”，而是在继续接你电脑上那条已经存在的会话。

## 5. 为什么我觉得它值得单独写一篇

前一篇写 harness engineering 的时候，`TermPilot` 更像那个落地对象。文章重点是我怎么把一个已经能跑的项目，整理成 agent 更容易进入、边界更稳定、验证更容易复跑的工程系统。

但如果项目本身没有先交代清楚，很多工程判断其实就失去了落点。

比如：

- 为什么要强调 `agent` 才是 session 的 source of truth
- 为什么 `relay` 只保存最小元数据
- 为什么要把 `shell` 和 `command` session 分开处理
- 为什么移动端只做查看、轻输入和轻控制

这些都不是抽象架构偏好，而是直接服务于产品边界的判断。因为 `TermPilot` 从一开始就不是远程桌面，也不是通用远控平台。它解决的问题更集中：让一条已经在电脑上运行的终端会话，可以被手机继续接上。

所以我觉得更合理的阅读顺序应该是：

1. 先把 `TermPilot` 是什么讲清楚
2. 再去看我是怎么围绕这条产品路径整理工程系统的

## 6. 当前边界

`TermPilot` 现在没有打算覆盖这些方向：

- 远程桌面
- 图形界面控制
- 手机上长时间重度终端编辑
- 自动接管任意已有 Terminal / iTerm 标签页
- 通用多租户运维平台

当前重点仍然是长期任务和会话连续性。

这个边界我倾向于保持明确。边界一旦不清楚，产品就很容易变成“什么都能沾一点，但主路径不够顺”。对一个个人项目来说，先把最核心的场景做顺，比同时解释太多扩展方向更重要。

## 7. 结语

如果你平时经常会在电脑上挂着一条长期任务，然后离开桌面之后还想继续看它、补一点轻控制，那 `TermPilot` 可能会比较适合你。

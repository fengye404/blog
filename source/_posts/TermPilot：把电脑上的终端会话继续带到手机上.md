# TermPilot：把电脑上的终端会话继续带到手机上

> 本文由 AI 参与创作

## 前言

`TermPilot` 是一套很小的端侧工具：`relay` 提供公网入口和手机端 Web 页面，`agent` 跑在你自己的电脑上，用 `tmux` 托管终端会话。手机浏览器连上来之后，看到的是**同一条**本地会话的输出，输入也会写回这条会话。

最近主要在家里写它。典型场景是：电脑上已经开着 `Claude Code`、部署脚本、迁移或批处理，人离开座位一会儿，任务还在跑。这时我在手机上要做的事情通常很少：扫一眼输出，确认没卡死，偶尔补一条命令，或者干净地中断。

这个项目从一开始就是按个人使用来收敛的：自己开发、自己部署、自己的手机接自己的机器。它不解决「团队协作」「多租户控制台」这类问题，只解决一条：**会话已经在跑了，怎样在另一块屏幕上接着用**。

## 1. 问题边界：要的是「同一条会话」，不是「再开一个 shell」

手机上的 SSH 客户端很多，登录到机器再开一个 shell 并不难。麻烦在于：**新 shell 里没有你已经跑了一半的那条上下文**。

目录、`env`、已经打印出来的日志、交互式工具卡在等确认——这些都已经绑在原来的 PTY 上。尤其是 `Claude Code` 这类流程，信任目录、确认对话框往往就发生在当前终端会话里。换一条会话，等于换了一个现场。

所以我想保留的能力可以压成一句：**输出和控制都锚在同一条本地受管会话上**。手机端不做「替代桌面终端」，只做「这条会话的第二块屏 + 轻量输入」。

桌面端进入 `Claude Code` 后，安全确认仍然发生在原来的终端上下文里：

![Claude Code 安全确认截图](./claude-trust-folder.png)

我比较在意的约束是：

- 输出来自同一条本地会话
- 控制动作写回同一条本地会话
- 会话结束语义跟着这条会话走（例如 `tmux` 里窗格关掉就是关掉）

这几条凑在一起，才配得上「继续」这两个字。

## 2. 最短使用路径

如果只是自己在家里用，链路可以压得很短：

```bash
npm install -g @fengye404/termpilot

# 一台手机能访问到的机器（需要公网可达的 HTTPS/WSS，见下文）
termpilot relay

# 你自己的电脑
termpilot agent --relay wss://your-domain.com/ws
termpilot claude code
```

不跑 `Claude Code` 时，也可以直接托管任意命令：

```bash
termpilot run -- <command>
```

第一次启动 `agent` 会打印设备信息和一次性配对码。手机浏览器打开 `relay` 的 Web 地址，输入配对码，完成与这台电脑的绑定。

![agent 启动与配对码截图](./agent-pairing.png)

配对完成后，手机端看到的是这台机器上**由 TermPilot 管理的那条会话**的实时视图：

![手机端会话截图](./mobile-session.png)

我自己最常用的就是这条路径：桌面负责重活和长会话，手机负责扫输出和偶尔插一句。

### 2.1 部署上需要提前想清楚的点

`relay` 必须放在手机能访问到的网络上，一般是公网机器或内网穿透后的等价入口。生产用法里 `agent` 和 `relay` 之间是 `wss://`，意味着你要有合法 TLS 证书和域名，文档站里有对应部署说明。

`agent` 侧除了 Node.js，还依赖本机可用的 `tmux`：会话附着、分离、回放都建立在 `tmux` 的行为上。若本机没有 `tmux` 或版本过旧，要先补齐环境。

更完整的 CLI、环境变量和排障步骤见文档站：<https://fengye404.top/TermPilot/>。

## 3. 信任与数据落点

配对码解决的是「哪台手机允许接哪台 agent」的初绑问题。绑定关系、授权记录和审计元数据由 `relay` 持久化；默认配置下长期运行用 SQLite 就够，不需要为了跑这个再去起一套重型数据库。

**会话标题、当前工作目录、终端滚动缓冲、运行状态**这些「重数据」留在 `agent` 所在机器。`relay` 做 HTTP 入口、WebSocket 中转和最小元数据存储，不当终端输出的主库。

这样拆的原因很朴素：

- 公网入口机器只暴露必要面，终端主数据不经过它落盘
- `agent` 才是会话的 source of truth，断网或 `relay` 重启不会把本地会话弄丢
- 手机端看到的是同步过来的视图，而不是在云端克隆了一套 shell

仓库里还有设备离线、会话退出、疑似残留会话等后台提醒，用来减少「以为还在跑其实早就断了」这类状态漂移。

## 4. 和 Happy 放在同一桌面对比

GitHub 上的 [Happy](https://github.com/slopus/happy) 已经做得很完整：面向 `Claude Code` / Codex 的移动与 Web 客户端，有 iOS、Android、Web、E2E 加密、推送、语音、多设备切换等。官方 [self-hosting 文档](https://happy.engineering/docs/guides/self-hosting/) 也写得很细。

Happy 自托管路径里要起 `happy-server`，配 `PostgreSQL` 和 `Redis`，Docker Compose 里常见依赖一起上，文档里还有企业网络和 Kubernetes 的延伸。它明显按「完整产品」在演进。

我做 `TermPilot` 的动机很简单：Happy 的能力我认，但在我自己的使用半径里过重。我想要的是：

- 手机端：移动 Web 即可
- 电脑端：一个 `agent` 进程
- 公网：一个 `relay`
- 持久化：`relay` 侧 SQLite 存配对与审计；会话本体在本地

形状更窄，部署心智负担更低，功能上也少做一层「移动协作产品」该有但我用不到的东西。两者不是替代关系，是半径不同的两种选型。

## 5. 当前实现结构

三个 runtime 分工如下：

- **`relay`**：HTTP 入口、配对与授权、审计元数据、托管移动端 Web、WebSocket 转发
- **`agent`**：本地 `tmux` 生命周期、输出回放、状态同步、与 `relay` 的长连接
- **`app`**：移动端页面，由 `relay` 直接提供静态资源

会话后端选 `tmux` 的原因前面已经说了：需要可附着、可分离、可长期存活的一条会话，而不是一次性 `exec` 跑完就散的命令执行器。

`relay` 启动示意：

![relay 启动截图](./relay-start.png)

有一条边界写死在设计里：**TermPilot 不会自动接管系统里已有的 Terminal / iTerm 标签页。**只有它创建或显式纳入管理的会话，手机端才能接。这是刻意收窄攻击面和状态来源，避免「扫全盘 PTY」这种不可控行为。

## 6. 结语

若你的场景接近下面几条，`TermPilot` 这种收得很窄的形态可能刚好够用：

- 主要在自己家或可控网络里用
- 桌面上已经有一条长会话在跑，离开座位时想用手机接着看
- 能接受自己维护 `relay` 入口和 TLS，不需要多租户控制台

核心仍是一句话：**把电脑上已经跑起来的那条终端会话，继续带到手机上。**
